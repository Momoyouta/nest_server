## Why

当前课程仅有“课程-老师”直接关系，难以支撑后续按教学组批量导入和分组管理；同时学生入课缺少面向课程与教学组的一次性邀请码能力，导致发码、追踪、失效管理不统一。管理端在绑定教学组前也缺少“按学校+姓名前缀”检索老师能力，且课程基础信息未返回邀请码时间信息，影响运营侧配置与核对效率。现在已新增教学组相关表与邀请码字段，需尽快补齐接口与契约，避免前后端并行开发受阻。

## Goals

- 建立“课程-教学组-老师”绑定链路，支持管理员为课程维护任课老师。
- 课程创建时自动创建默认教学组，保证后续绑定与导入可复用。
- 提供教学组独立 CRUD 能力，支持前端按课程进行教学组配置。
- 教学组老师绑定后，课程老师关系表 `course_teacher` 同步维护对应记录。
- 提供按学校与姓名前缀查询老师（返回老师 ID 与姓名）的接口，支撑教学组绑定选人。
- 提供不限次数的课程邀请码生成能力，使学生可通过邀请码加入指定学校课程的指定教学组。
- 学生重复使用邀请码加入已选课程时，明确提示“已加入该课程”。
- 扩展课程基础信息查询，仅返回未过期的邀请码及其创建时间、持续时间（时间戳语义）。
- 统一邀请码在 MySQL+Redis 的记录、校验与过期清理约束。

## Non-goals

- 本次不实现教师批量导入，仅预留可复用服务能力。
- 本次不改造学生端完整入课流程 UI，仅定义后端接口与行为契约。
- 本次不扩展非课程场景的邀请码类型。

## What Changes

- 新增管理端课程教学组与任课老师绑定接口（含可复用服务层）。
- 新增管理端教学组 CRUD 接口（创建、列表、详情、更新、删除），详情/更新/删除统一基于 `teaching_group_id`。
- 新增本校老师检索接口：基于 `school_id + name` 前缀匹配查询老师 `id/name`，用于教学组绑定。
- 调整课程创建流程：创建课程后自动创建教学组。
- 新增课程邀请码生成接口（管理员/同校老师可发码），邀请码绑定 school_id、course_id、teaching_group，且不限制使用次数。
- 教师发码权限收敛为“必须已绑定到目标课程教学组”。
- 扩展 `getCourseBasicAdmin`：仅返回未过期邀请码及其创建时间、持续时间字段。
- 调整学生使用邀请码入课行为：重复加入时返回“已加入该课程”提示。
- 更新邀请码持久化与缓存策略：MySQL 记录 + Redis 存储 + 定时清理。
- 更新 OpenAPI 3.0 文档与 DTO，明确鉴权、请求参数、返回结构。

## Capabilities

### New Capabilities
- `course-teaching-group-management`: 管理课程下教学组、教学组老师绑定与可复用绑定逻辑。
- `course-invitation-code`: 生成用于学生加入课程教学组的邀请码，并定义缓存与过期清理行为。

### Modified Capabilities
- `admin-course-management`: 课程创建成功后必须自动初始化教学组，并在课程管理域补充老师检索与课程基础信息邀请码返回能力。

## Impact

- 数据库实体：course_student、course_teaching_group、course_group_teacher、invitaion_code 需按真实表结构更新。
- 模块与接口：course、invitation、student 相关 service/controller/DTO 与 Swagger 注解。
- 查询行为：课程管理新增老师前缀检索接口，课程基础信息增加邀请码时间戳字段。
- 关系一致性：教学组老师绑定时需同步维护 `course_teacher` 课程老师关系。
- 基础设施：Redis key 设计、邀请码过期与定时清理任务。
