# 提案：邀请码系统

## 动机 (Why)
平台需要一种受控的方式让教师和学生加入学校。目前注册是开放的，但需要限制为只有持有由授权用户（管理员）生成的有效邀请码的人才能注册。

## 目标 (Goal)
- **管理员管理**：允许管理员创建、删除和分页搜索邀请码。邀请码包含特定元数据（类型、学校、年级/入学年份、班级、创建者、时限 TTL、时间戳）。
- **持久化存储**：邀请码信息将持久化在 MySQL 数据库中以便于复杂查询，并可同步至 Redis 用于快速验证。
- **注册流程**：
  - 增强注册过程，要求非管理员用户提供并验证邀请码。
  - 根据邀请码内容，自动将新用户与正确的实体（学校、教师/学生）关联。
- **基于角色的逻辑**：
  - 如果教师凭有效邀请码注册，则创建 `User` 和 `Teacher` 实体。
  - 如果学生凭有效邀请码注册，则创建 `User` 和 `Student` 实体，并将其与指定的年级（入学年份）/班级关联。

## 非目标 (Non-goal)
- **课程邀请**：邀请学生加入特定课程不在本次迭代范围内。
- **管理员邀请**：目前不需要为新管理员创建邀请码。

## 影响范围 (Affected Areas)
- `src/common/dto`：更新邀请码 DTO 结构，由 `grade_id` 改为 `grade`，新增查询 DTO。
- `src/database/entities`：新增 `InvitationCode` 实体。
- `src/modules/auth`：处理带邀请码注册的 `AuthController` 和 `AuthService`。
- `src/modules/user`：处理学生/教师实体创建的 `UserService`。
- `src/modules/invitation`：实现邀请码的创建、删除、分页查询功能。
- `src/common/utils/invite.util.ts`：仍用于代码生成和 Redis 相关操作。

## 影响 (Impact)
此更改将使平台从开放注册模型迁移到许可入驻模型，通过从注册时刻起将用户与其各自的学校和班级正确链接，确保数据完整性。
